Overview

Cryptocurrency markets are characterized by high-frequency price fluctuations, fragmented liquidity, and rapid regime shifts, making real-time volatility detection a relevant problem for both market monitoring and risk-aware decision support. This project addresses that challenge by developing a production-oriented MLOps pipeline that continuously ingests live Coinbase market data, transforms raw tick-level events into short-window statistical features, and serves a binary volatility signal through a deployable machine learning API.

The system integrates the full operational machine learning lifecycle, including streaming data ingestion with Kafka, feature engineering over rolling 60-second windows, model training and experiment tracking with XGBoost and MLflow, REST-based inference through FastAPI, and observability through Prometheus, Grafana, and drift monitoring. By combining real-time data processing, reproducible model management, and deployment-level monitoring, the project demonstrates how financial machine learning models can be operationalized beyond offline experimentation.

Architecture

Ingest layer: Kafka

A Python producer connects to the Coinbase Advanced Trade WebSocket API and streams raw ticker events into a Kafka topic (ticks.raw). Raw records are persisted as NDJSON for reproducibility and replay. A downstream consumer processes those events and writes labeled feature sets to Parquet files, cleanly separating ingest from feature computation.

Feature engineering

Features are derived exclusively from current and historical data, with no lookahead. They are organized into three groups:

  • Price-derived: midprice returns, rolling return averages across four window sizes (5, 20, 60, 100 observations), realized volatility, volatility-of-volatility, and return z-scores normalized by recent volatility
  • Liquidity: bid-ask spread in basis points, spread changes, spread-normalized pressure, rolling spread statistics, and trade intensity across three time windows (60s, 300s, 900s)
  • Regime/state: cyclical encodings for time-of-day and day-of-week, capturing intraday and weekly market behavior patterns

The binary label is a volatility spike indicator: 1 if future realized volatility over the next 60 seconds exceeds the 90th-percentile threshold for that product, 0 otherwise.

Model training: XGBoost + MLflow

Two model variants are supported and swappable via the MODEL_VARIANT environment variable:

  • Baseline: a conservative rule-based classifier, recommended for production stability
  • ML: an XGBoost binary classifier with stronger precision-recall trade-offs

All training runs, hyperparameter sweeps, and evaluation metrics (AUC-ROC, F1, precision/recall) are tracked in MLflow, with model artifacts registered in the MLflow Model Registry for version-controlled serving.

Serving: FastAPI

A FastAPI application loads the registered model at startup and exposes four endpoints: /predict (accepts a feature row and returns a volatility score and binary label), /health, /version, and /metrics for Prometheus scraping.

Observability

Prometheus scrapes the /metrics endpoint and Grafana dashboards surface prediction latency, request throughput, and rolling accuracy. Evidently runs a separate drift detector that compares the live feature distribution against the training reference, raising alerts when distribution shift is detected.

One-command deployment

All five services (Zookeeper, Kafka, FastAPI, Prometheus, and Grafana) are declared in a single docker-compose.yml. The entire pipeline stands up on any machine with: docker compose up --build.

How it was built

  • Phase 1

    Kafka (KRaft) and MLflow launched via Docker Compose. WebSocket producer ingesting live Coinbase ticker data into ticks.raw, with reconnect and heartbeat logic. Scoping brief written defining the 60-second prediction goal.

  • Phase 2

    Featurizer built as a Kafka consumer: computed windowed features and wrote output to Parquet. EDA conducted in a notebook; spike threshold set using percentile plots. First Evidently report generated comparing early and late data windows.

  • Phase 3

    Baseline (z-score rule) and XGBoost models trained on time-based splits. All runs logged to MLflow with PR-AUC as the primary metric. Model card and GenAI appendix written.

  • Phase 4

    FastAPI serving layer built with /predict, /health, /version, and /metrics endpoints. Full Docker Compose stack assembled and tested in replay mode.

  • Phase 5

    CI pipeline set up with GitHub Actions (lint + integration test). Reconnect, retry, and graceful shutdown added to Kafka services. Load test run at 100 burst requests.

  • Phase 6

    Prometheus metrics instrumented (latency, request count, errors, consumer lag). Grafana dashboards built. SLOs defined, Evidently drift summary scheduled, and MODEL_VARIANT rollback toggle added.

  • Phase 7

    Demo recorded showing startup, live prediction, failure recovery, and model rollback. Runbook and final documentation finalized. Repository tagged for release.
