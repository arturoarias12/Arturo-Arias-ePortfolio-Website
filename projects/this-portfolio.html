Why build it from scratch?

Most portfolio templates come with a framework you did not choose, opinions you did not form, and constraints you have to fight. Building from scratch meant every layout decision, every color token, and every interaction was deliberate.

The goal was not to demonstrate that I can use a site builder. The goal was to produce a site that reflects how I actually think about design and engineering — precise, readable, fast, and maintainable by one person.

Design decisions

CSS custom properties as the design system

All colors, radii, spacing units, and transition timings live in :root as CSS variables. Dark mode is implemented by swapping the values at the [data-theme="dark"] selector level — no JavaScript class toggling, no duplicated style blocks.

css
:root {
  --primary: rgb(0, 0, 190);
  --bg: #ffffff;
  --text: #0a0a3a;
  --text-muted: #5f5e5a;
}
[data-theme="dark"] {
  --primary: rgb(41, 98, 255);
  --bg: #050510;
  --text: #e8e8f5;
  --text-muted: #b0bcdd;
}

Sticky avatar navbar

The navbar is sticky with a frosted-glass background (backdrop-filter: blur), collapses to a hamburger at 820px, and uses a circular avatar image as the brand mark. All scroll-linked behaviors use a single scroll event listener with passive: true.

Projects system

Projects are defined in a single projects-data.js file as a plain JavaScript array. The listing page (projects.html) reads this array and renders cards dynamically, with client-side search, category filters, tag filters, sort, and grid/list view toggle — no server, no build step, no framework.

Google Apps Script contact form

The contact form posts to a Google Apps Script endpoint using fetch with Content-Type: text/plain (required to avoid a preflight CORS request). The script reads the JSON payload, validates it, and emails the submission to my inbox. A honeypot field is included to block naive bots.

javascript
fetch(SCRIPT_URL, {
  method: 'POST',
  headers: { 'Content-Type': 'text/plain' },
  body: JSON.stringify({ name, email, message })
})
.then(r => r.json())
.then(data => {
  if (data.result === 'success') showSuccess();
  else showError();
});

AI chat assistant

Every page on the site includes a floating AI chat assistant that answers questions about Arturo's background, projects, experience, and skills. The widget is a self-contained vanilla JavaScript IIFE injected via a shared script. The intelligence lives entirely server-side in a Cloudflare Worker.

Architecture

The frontend sends a single { message } field to the Worker API. The Worker performs RAG retrieval, builds the prompt, calls the LLM, and returns { answer }. No API keys or system prompts are exposed to the browser.

RAG pipeline (Cloudflare KV)

The knowledge base is 20 Markdown files covering every section of the site: about, experience, education, skills, certifications, all 12 projects, blog, and contact. Each file is stored as an individual entry in a Cloudflare KV namespace. An index.json maps filenames to titles and keyword arrays.

For each request the Worker loads the index, scores each document against the query using title and keyword overlap, fetches the top matching documents from KV, and injects their content into the system prompt. This keeps the LLM grounded in site-specific facts without fine-tuning the model.

LLM inference via Cloudflare Workers AI

The language model runs on Cloudflare's global edge inference infrastructure through the env.AI binding. No external API key is required. The system prompt is stored as an encrypted Cloudflare environment variable — it never appears in source code or version control.

javascript
const ai = await env.AI.run('@cf/meta/llama-3.1-8b-instruct-fp8-fast', {
  messages: [
    { role: 'system', content: systemPrompt + ragContext },
    { role: 'user',   content: message },
  ],
  max_tokens: 512,
  temperature: 0.5,
});
return Response.json({ answer: ai.response });

Security and rate limiting

Rate limiting is enforced at the Cloudflare WAF level, with a block action applied per IP on excess requests. The Worker additionally validates HTTP method, Content-Type, JSON structure, message length, and message count before any inference call is made. The system prompt includes explicit instructions against prompt injection, jailbreaking, and out-of-scope requests.

Custom domain

The chatbot API is served from a dedicated Cloudflare custom subdomain wired to the Worker. The portfolio itself is hosted on GitHub Pages under arturoarias.me, managed through Cloudflare DNS.

Reflections

The hardest part was not the code — it was the copy. Every word on this site required deciding what to say and what to leave out. Building the design system took an afternoon; editing the About page took three days.

If I were starting over: I would define the CSS token system before writing a single HTML element. I refactored the color tokens twice after painting myself into a dark-mode corner with hard-coded values.
